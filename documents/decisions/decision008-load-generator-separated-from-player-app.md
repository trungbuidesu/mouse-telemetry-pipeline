# Decision DEC-008: Load Generator Separated From Player App

## 1. Status

`DECIDED`

Date: `2026-07-22`

---

## 2. Context

The pipeline may need more events than one real player can comfortably generate. Increasing raw mousemove frequency in the player app would hurt UX and weaken the performance design. The architecture explicitly suggests using a separate load generator for larger pipeline load.

---

## 3. Decision

Synthetic load must be generated by a separate script/tool that sends valid session and telemetry batches through the same FastAPI ingestion API.

The player app keeps the MVP sampling and batching defaults from DEC-002.

---

## 4. Rationale

- Player UX stays representative and responsive.
- Load testing becomes configurable and repeatable.
- Synthetic data still exercises FastAPI, Kafka, Spark and storage.
- Demo can clearly separate manual session behavior from stress/load behavior.

---

## 5. Consequences

### Positive

- Controlled throughput for dashboard demo.
- Safe defaults for local machine.
- No need to distort frontend telemetry hot path.

### Trade-offs

- Additional script/tool must be maintained.
- Synthetic data should be marked or documented to avoid confusing it with manual play.

---

## 6. Implementation Constraints

- Load generator must send HTTP batches, not direct Kafka messages.
- It should generate `session_start`, `mousemove`, `click` and `session_end` events.
- It should expose session count, duration, event rate and batch size controls.
- Defaults must avoid overwhelming local Docker Compose services.
